# 一键回滚方案

---

## 📋 方案概述

本方案提供生产环境一键回滚能力，涵盖**数据库、服务、流量**三个核心维度，确保在出现严重问题时能够快速恢复到上一个稳定版本。

### 🎯 设计目标

| 目标 | 说明 |
|------|------|
| **快速回滚** | 整体回滚时间 < 5分钟 |
| **数据安全** | 数据库采用备份恢复，无数据丢失 |
| **流量无损** | 流量先切回稳定版本，再回滚服务 |
| **验证闭环** | 回滚后自动验证，确保服务正常 |

## 📝 回滚步骤详解

### 步骤1: 流量回滚

```bash
# 检查当前流量分配
kubectl get ingress ai-director-canary

# 将流量切回稳定版本
kubectl apply -f k8s-rollout/ingress/ai-director-stable.yaml
```

**目的**: 在回滚服务前，先将用户流量引导至稳定版本，确保用户体验不受影响。

### 步骤2: 服务回滚

```bash
# 回滚到稳定版本
kubectl set image deployment/ai-director \
  ai-director=registry.example.com/ai-director:v1.9.0

# 等待部署完成
kubectl rollout status deployment/ai-director --timeout=120s
```

**目的**: 将服务回滚到上一个经过验证的稳定版本。

### 步骤3: 数据库回滚

```bash
# 停止数据库写入
kubectl scale deployment/ai-director --replicas=0

# 恢复备份
cp backups/dev.db.backup_20260615_112754 /var/lib/sqlite/dev.db

# 重启服务
kubectl scale deployment/ai-director --replicas=3
```

**目的**: 将数据库恢复到回滚前的状态，确保数据一致性。

## ⚠️ 自动回滚触发条件

当监控系统检测到以下任一条件时，自动触发回滚：

| 指标 | 阈值 | 动作 |
|------|------|------|
| 错误率 | > 1% | 停止扩容 → 自动回滚 |
| P95延迟 | > 500ms | 自动回滚 |
| CPU使用率 | > 70% | 告警 |
| 内存使用率 | > 75% | 告警 |
| 超时率 | > 5% | 自动回滚 |

---

## 📊 回滚验证检查项

| 检查项 | 检查方法 | 成功条件 |
|--------|----------|----------|
| 服务副本数 | `kubectl get deployment` | readyReplicas == replicas |
| 版本号 | `kubectl get deployment -o jsonpath` | 等于 v1.9.0 |
| 健康检查 | `curl /api/v1/health` | HTTP 200 |
| 数据库连接 | `curl /api/v1/health` | database 状态正常 |
| 流量配置 | `kubectl get ingress` | backend 指向 stable |

## 📋 前置条件

1. ✅ 已创建数据库备份（`backups/`目录下）
2. ✅ Kubernetes集群正常运行
3. ✅ kubectl命令可用且有权限
4. ✅ 稳定版本镜像已存在于镜像仓库

---

## 🚨 注意事项

1. **数据库回滚会覆盖当前数据**，请确保在执行前已经确认需要回滚
2. 回滚过程中服务会短暂不可用（约30秒）
3. 建议在低峰期执行回滚操作
4. 回滚后建议进行一次完整的功能测试
